# In-App Credit Purchase System — Design Spec

## Goal

Allow VisionPipe desktop app users to purchase credit packs directly from the app using Stripe Checkout, without requiring a login/auth system. Credits are tied to a device ID.

## Architecture

Device ID (UUID) generated on first app launch, persisted locally via `tauri-plugin-store`. The Next.js web app (`visionpipe-web`) provides API routes for device registration, Stripe Checkout session creation, webhook fulfillment, and balance queries. Vercel Postgres stores device balances.

Two repos are involved:
- `VisionPipe/visionpipe` — Tauri desktop app (device ID, balance sync, purchase UI)
- `VisionPipe/visionpipe-web` — Next.js web app (API routes, Stripe integration, database)

## Component 1: Database (visionpipe-web)

### Schema

One table in Vercel Postgres:

```sql
CREATE TABLE devices (
  id UUID PRIMARY KEY,
  balance INTEGER NOT NULL DEFAULT 0,
  email TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

- `id`: Device UUID generated by the Tauri app
- `balance`: Credit balance (server is source of truth)
- `email`: Nullable, for future account linking / receipt emails
- `created_at` / `updated_at`: Timestamps

### Why one table

No transaction history table for v1. The Stripe Dashboard provides full payment history. A `transactions` table can be added later if needed for in-app history display.

## Component 2: API Routes (visionpipe-web)

All routes live under `src/app/api/` in the Next.js App Router.

### POST /api/devices/register

Called on first app launch to register the device.

- **Request body**: `{ "deviceId": "uuid-string" }`
- **Response**: `{ "deviceId": "...", "balance": 0 }`
- **Behavior**: INSERT device row. If device already exists (conflict on `id`), return existing balance. This makes the endpoint idempotent — safe to call on every app launch.

### GET /api/balance?deviceId=xxx

Called by the app to sync credit balance.

- **Query param**: `deviceId` (UUID)
- **Response**: `{ "balance": 1234 }`
- **Behavior**: SELECT balance from devices table. 404 if device not found.

### POST /api/checkout

Called when user clicks "Buy Credits" to create a Stripe Checkout Session.

- **Request body**: `{ "deviceId": "uuid-string", "packId": "starter" | "pro" | "business" }`
- **Response**: `{ "url": "https://checkout.stripe.com/..." }`
- **Behavior**:
  1. Look up the Stripe Price ID for the requested pack
  2. Create a Stripe Checkout Session with:
     - `mode: "payment"` (one-time)
     - `metadata: { deviceId, packId, credits }` (for webhook fulfillment)
     - `success_url`: redirect back to a thank-you page or `visionpipe://credits-added`
     - `cancel_url`: redirect back to pricing or close
  3. Return the checkout URL

### POST /api/stripe/webhook

Called by Stripe when payment events occur. Already configured at `https://api.visionpipe.ai/stripe/webhook`.

- **Handles**: `checkout.session.completed`
  1. Verify webhook signature using Stripe signing secret
  2. Extract `deviceId` and `credits` from session metadata
  3. `UPDATE devices SET balance = balance + $credits, updated_at = NOW() WHERE id = $deviceId`
  4. Return 200

- **Handles**: `checkout.session.expired`
  1. Log for observability
  2. Return 200 (no action needed)

### Security

- Webhook signature verification via `stripe.webhooks.constructEvent()`
- Device ID is a UUID — not guessable, but not a secret. Balance queries are read-only and low-risk.
- No auth tokens needed for v1. If abuse becomes a concern, add rate limiting.

## Component 3: Device ID and Balance Sync (visionpipe app)

### Device ID Generation

In `src-tauri/src/credits.rs`, add:
- On first launch, generate a UUID v4 and store it in `tauri-plugin-store` under key `device_id`
- On subsequent launches, load the existing device ID
- Call `POST /api/devices/register` on every launch (idempotent) and sync the returned balance

### Balance Sync

New Tauri command `sync_credits` that:
1. Reads `device_id` from store
2. Calls `GET /api/balance?deviceId=xxx`
3. Updates the local `CreditLedger` balance
4. Returns the new balance to the frontend

Called:
- On app launch (after device registration)
- After returning from Stripe Checkout (poll until balance changes or timeout)
- When the user manually refreshes

### Deduction: Local-First with Server Sync

Keep the current local deduction flow (`deduct_credits` command). After a local deduction:
1. Deduct locally (immediate UX feedback)
2. Call `POST /api/deduct` (we need to add this endpoint) to sync the deduction server-side
3. If server deduction fails (e.g., server balance is lower), re-sync from server

This means the local balance is a cache. Server is the source of truth.

### New API endpoint needed: POST /api/deduct

- **Request body**: `{ "deviceId": "uuid-string", "amount": 5 }`
- **Response**: `{ "balance": 1229 }` or `{ "error": "insufficient_credits", "balance": 2 }`
- **Behavior**: Atomic `UPDATE devices SET balance = balance - $amount WHERE id = $deviceId AND balance >= $amount`. If 0 rows updated, return error with current balance.

## Component 4: Purchase UI (visionpipe app)

### "Buy Credits" Button

In the sidebar credit section of `src/App.tsx`, when balance is low or user clicks:
- Show a simple pack selector (3 options: Starter / Pro / Business with prices)
- On selection, call a new Tauri command `start_checkout` that:
  1. POSTs to `/api/checkout` with `deviceId` and `packId`
  2. Opens the returned URL in the system browser via `tauri-plugin-shell`
- After the browser opens, start polling `sync_credits` every 3 seconds for up to 2 minutes to detect when credits arrive

### UI Changes

- Remove the `app = visionpipe` box from sidebar
- Remove the `context` header (keep the context input box)
- Add "Buy Credits" link/button near the balance display
- Show pack selection inline or as a small modal
